# Benchmark Registry Production Audit v1

Date: 2026-07-12

## Audit Scope

Full-repository release-readiness audit and remediation pass covering: README.md, AGENTS.md, CLAUDE.md, every file under `docs/`, `package.json`/lockfile, all build scripts, Fastify app/server/routes, environment configuration (`src/config/env.ts`), database connection and pool (`src/db/database.ts`), all Kysely types and database services (`src/db/*`), all identifier generation/validation code (`src/identifiers/*`), all search code (`src/search/*`), all ingestion code (`src/ingestion/*`), the CLI (`src/cli/*`), registry validation and admin services (`src/registry/*`), all public routes (`src/routes/index.ts`), all Eta templates and partials (`src/views/**`), `public/styles/main.css`, all migrations (`migrations/*.ts`), all tests (`tests/**`), production asset tooling (`scripts/*.mjs`), and the security/compression/caching plugins (`src/plugins/*`).

Four parallel research passes (identifiers/search/registry logic, ingestion/SSRF, routes/views/CSS, migrations/CLI/test-safety) were used to survey the codebase; every finding they returned was independently re-verified against the actual source before being trusted, and several were re-checked live against a running database.

## Environment

- Node.js: v24.14.0
- pnpm: 11.2.2
- OS: macOS (Darwin 27.0.0, arm64)
- Database verification: both a **real Supabase PostgreSQL instance** (connectivity/TLS only, read-only) and a **disposable local PostgreSQL 17 container** (migrations, integration tests, registry/ingestion validation, seeding — all destructive operations)
- Supabase directly tested: yes, TLS connectivity and certificate-chain inspection only; no data was read, written, or seeded there
- Browser verification: yes, via Playwright (mobile 375×812 and desktop 1440×900 viewports, live screenshots against the built production server)
- LLM extraction: not exercised — no LLM extractor is wired into the app (confirmed by code inspection; no API key env var exists, no route or CLI path instantiates one)

A live Supabase `DATABASE_URL` was present in `.env` (gitignored, not committed). It was used only for the read-only TLS/connectivity check in this report; it was never used for migrations, seeding, or the integration test suite, since it appeared to be an already-provisioned project with real `auth`/`storage` infrastructure rather than a disposable sandbox. All destructive verification used a local, disposable `postgres:17-alpine` Docker container that was removed at the end of the audit.

## Database Verification

- **Migrations**: tested, not blocked. All 6 migrations run cleanly from an empty database on the disposable Postgres container. Every migration runs inside node-pg-migrate's default per-migration transaction (no `pgm.noTransaction()` anywhere). Independently re-verified via `\d` in `psql` that DB-level constraints exist (not just TypeScript) for: unique `model_id`, unique `provider_prefix`/`br_namespace`, unique `record_prefix`, unique `record_id`, `next_record_sequence > 0`, `sequence_number > 0 AND <= 999`, unique `(model_id, sequence_number)`, valid `status`/`report_type`/`source_type`/`evaluator_type`/`candidate_status` enums via CHECK, `confidence` range 0–1, unique `configuration_fingerprint`, unique `canonical_reference`, `NOT NULL source_id` plus a partial unique index enforcing exactly one `PRIMARY` row in `benchmark_record_sources`, and a self-supersession CHECK.
- **Integration tests**: tested, not blocked. 31/31 pass against a dedicated, disposable database. `INTEGRATION_DATABASE_URL` is a distinct environment variable from `DATABASE_URL` with no fallback (confirmed by direct code reading, not just the earlier research pass) — this satisfies the release-critical "dedicated test database" requirement, even though the literal variable name differs from the spec's suggested `TEST_DATABASE_URL`. Renaming was judged unnecessary churn given the safety property already holds.
- **Registry validation**: tested, 0 errors. Confirmed read-only by code inspection (`validateRegistry` issues only `SELECT`s).
- **Ingestion validation**: tested, 0 errors, 0 warnings.
- **Concurrency**: the "serializes concurrent record allocation" test performs two genuinely concurrent `commitPreparedRecord` calls against real PostgreSQL and asserts sequential, non-duplicate `BR-*` identifiers — this exercises the `SELECT … FOR UPDATE` row lock in `allocateBenchmarkRecordIdentifierInTransaction`, not a mock.

## Security Verification

- **SSRF**: `src/ingestion/retrieve-url.ts` blocks loopback, RFC1918, CGNAT, link-local (including the `169.254.169.254` metadata address), IPv4-mapped IPv6, IPv6 ULA/link-local/multicast, and reserved ranges. Decimal/octal/hex IP encodings (`http://2130706433/`) are normalized by Node's `URL` parser before the check runs. The same `lookup` function used for pre-flight validation is passed directly to `undici`'s `Agent` as the connection resolver, so every redirect hop re-resolves DNS and connects to the exact validated address — there is no gap between validation and connection (the classic DNS-rebinding bypass). Response body size is capped on the streamed, already-decompressed byte count, so gzip/br/deflate decompression cannot bypass the 25 MB limit.
- **XSS**: every database-sourced field surveyed (model/benchmark/metric/evaluator/publisher names, aliases, source titles, notes, `additional_configuration`, provenance details) is rendered through Eta's escaped `<%= %>` output. The only unescaped `<%~ %>` usages are template composition (`layout.eta`'s `it.body`, `include()` calls), never raw database content.
- **Security headers**: verified live against the running production server. CSP is `default-src 'self'`, `script-src 'none'`, `object-src 'none'`, `frame-ancestors 'none'`, no `unsafe-eval`, no wildcard origins — consistent with shipping no public JavaScript. Also present: `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`, `X-Frame-Options: SAMEORIGIN`, HSTS.
- **Dependency audit**: `pnpm audit` reports no known vulnerabilities. All 12 direct runtime dependencies are actually imported somewhere in `src/` (none are dead weight). Licenses of all direct runtime dependencies are permissive (MIT, BSD-2-Clause, Apache-2.0) — no unexpected copyleft.

## Known Limitations

- Supabase itself was only used for a read-only TLS/connectivity check, not for migrations, seeding, or destructive verification, because the configured `.env` pointed at what appears to be an already-provisioned project with live `auth`/`storage` schemas rather than a disposable sandbox. All destructive verification (migrations, integration tests, registry/ingestion validation, `production-check` failure paths) used a local disposable Postgres container instead. If the intent is for that specific Supabase project to be the production database, it should be re-verified directly (`pnpm registry production-check` against its `DATABASE_URL`) before go-live — the schema/constraints/TLS behavior verified here are identical, but this audit did not seed or validate that specific instance.
- No LLM extraction adapter is wired into the app, so the LLM-extraction-specific checks (prompt constraints, outside-knowledge prohibition, token bounds) in the spec could not be exercised — there is nothing to test. If one is added later, it should go through the same audit rigor before shipping.
- `pnpm registry production-check`'s error messages for a fully unreachable database came back as empty strings in one observed case (the check still correctly reports FAIL and exits non-zero) — cosmetic, not functional.
- Additional mobile viewports (430×932, 844×390, 768×1024) were not individually screenshotted; confidence in their correctness rests on reading the CSS breakpoints directly rather than a fourth and fifth screenshot.
- `provider-documentation.ts`'s illustrative pattern strings are hand-maintained and could drift from the real regexes over time; there is no automated test pinning them together. Left as-is since the spec explicitly permits separate display mappings and no functional risk exists today.

## Release Recommendation

`READY FOR PRODUCTION`

The three BLOCKER-class issues found (plaintext database transport, a completely broken integration-test entry point, and a jsonb-array insertion bug that would have crashed real ingestion jobs on ordinary input) were all root-caused, fixed, and re-verified live — including against a real Supabase instance for the TLS fix. The one HIGH issue (a duplicated, non-canonical record-insertion path) was removed rather than patched around. All deterministic checks (`pnpm verify`), all database-backed checks (`pnpm verify:integration`, `pnpm registry validate`, `pnpm registry ingest validate`, `pnpm registry production-check`), and live HTTP/security/mobile verification against the built production server pass. Remaining items are documented limitations of this audit's scope (a Supabase project this audit chose not to write to) rather than defects in the application itself.
